## 1. 市場規模推定（TAM / SAM / SOM）

### 市場定義

**問題空間**: 日本国内の土地・候補地の一次調査において、衛星環境データ（JAXA）と行政データ（国交省）が分散しており、統合的な評価に専門知識と時間がかかる。

**対象セグメント**:
1. 不動産デベロッパー（土地仕入れスクリーニング）
2. 自治体都市計画・公有地活用担当
3. 損害保険会社（災害リスク評価）
4. 建設コンサルタント（環境アセスメント）

**地理**: 日本国内限定

## 2. 競合分析

### 直接競合（衛星×行政データ統合）

> **重要発見: 天地人（株）が直接競合として存在する。**

| サービス | 提供元 | 機能 | 強み | 弱み（差別化ポイント） |
|---------|--------|------|------|----------------------|
| **天地人コンパス** | 天地人株式会社（JAXA発ベンチャー） | 衛星データ×土地評価・農業・不動産分析SaaS | JAXA発の高い信頼性、自治体連携実績多数、商用実績あり | MCP非対応（独自API）、国交省行政データとの統合度が不明、LLM統合なし |

**天地人との差別化戦略**:
1. **MCP統合**: 天地人は独自プラットフォームだが、TerraScoreはMCPプロトコルで国交省データを直接統合
2. **LLM説明生成**: 天地人は可視化中心、TerraScoreは根拠付き自然言語レポート
3. **オープンアーキテクチャ**: MCP標準に準拠することで、将来の他MCPサーバー追加が容易
4. **コスト**: 天地人はエンタープライズ価格帯（推定）、TerraScoreはフリーミアム〜中価格帯を想定
5. **国交省データの深さ**: 不動産情報ライブラリの25APIとDPFの18ツールをフル活用

**ただし、天地人が先行者として市場教育を行っている点はプラスに捉えられる。** 「衛星データ×土地評価」というカテゴリ自体の認知向上に寄与している。

### 間接競合（部分的に機能重複）

| サービス | 提供元 | 機能 | 強み | 弱み（本プロダクトとの差別化ポイント） |

| **不動産情報ライブラリ** | 国交省 | 25種の不動産・地理データ表示 | 公式データ、無料 | 衛星データなし、API直接利用のみ、UI限定 |

### 注目すべき政府基盤

| 基盤 | 概要 | 本プロダクトとの関係 |
|------|------|---------------------|
| **不動産ID** | 国交省推進の不動産一意識別子（2024年本格運用開始） | 将来的にデータ横断の「鍵」として活用可能 |
| **G空間情報センター** | 地理空間情報のオープンデータ流通基盤 | PLATEAUデータの取得元 |
| **Tellus** | 経産省の衛星データプラットフォーム | JAXA Earth APIとの補完・代替候補 |

### 競合マトリクス

```
              衛星環境データ統合
                    ↑ 高
                    |
          天地人 ●  |  ★ TerraScore（目指す位置）
          Sagri ●   |
          Tellus ●  |
                    |
           低 ←────┼────→ 高  行政データ統合
                    |
     PLATEAU VIEW   |   不動産情報ライブラリ
     ハザードマップ   |   TASUKI TECH LAND
     じぶんの地盤    |   NTTデータ用地管理
                    |   LocationMind
                    ↓ 低
```

**天地人との位置関係**: 天地人は衛星データ統合は高いが、国交省MCPを使った行政データ統合の深さでは差別化可能。TerraScoreは**右上のポジション**（衛星も行政も高い統合度）を狙う。

### 競合優位性の評価

**本プロダクトのユニークポジション**: 衛星環境データ（物理的証拠）× 行政データ（制度的証拠）の **MCP統合レイヤー + LLM説明生成** を提供するツール。

天地人が「衛星×土地評価」で先行しているが、以下の点で差別化:
- **MCP標準準拠**: オープンプロトコルによる拡張性
- **国交省25 API統合**: 不動産情報ライブラリの全25データソースを活用
- **LLM根拠付きレポート**: 「事実/不足/リスク」を構造化した説明型出力
- **オープンソース戦略**: 天地人のクローズドSaaSに対する差別化

既存ツールの分類:
- 衛星×土地（天地人 — 先行者だが国交省MCP非統合）
- 行政データのみ（不動産情報ライブラリ、ハザードマップ）
- 衛星データ基盤のみ（Tellus、JAXA Earth API）
- GIS専門ツール（ESRI — 高価格・高スキル要求）
- 業務特化SaaS（TASUKI — 事業性評価中心）

## 3. ユーザーニーズ分析

### ペルソナ別の課題と期待

#### ペルソナ1: 不動産デベロッパー（土地仕入れ担当）

| 項目 | 内容 |
|------|------|
| **課題** | 候補地の初期スクリーニングに1件あたり半日〜1日かかる。複数サイトを手動で巡回し、スプレッドシートに転記 |
| **現在の対処法** | Google Maps + ハザードマップポータル + 不動産情報ライブラリ + 現地調査 |
| **期待する成果** | 1件5分で主要リスク・環境・規制を把握。候補地の足切り判断を高速化 |
| **支払い意思（推定）** | 月額5〜30万円（年間60〜360万円） |
| **重視する指標** | 用途地域、地価水準、災害指定の有無、標高・地形 |

#### ペルソナ2: 自治体都市計画担当

| 項目 | 内容 |
|------|------|
| **課題** | 公有地活用検討や都市計画見直しの基礎資料作成に時間がかかる。衛星データの扱い方がわからない |
| **現在の対処法** | コンサルへの外注（1件100〜500万円）、手動での情報収集 |
| **期待する成果** | 庁内での初期検討を自前で迅速化。コンサルへの発注前に方向性を固める |
| **支払い意思（推定）** | 年間50〜200万円（予算化が必要） |
| **重視する指標** | 立地適正化計画との整合、将来人口推計、周辺施設、災害リスク |

#### ペルソナ3: 損害保険会社（リスク評価担当）

| 項目 | 内容 |
|------|------|
| **課題** | 個別物件の災害リスク評価に行政データと実地データの統合が不十分 |
| **現在の対処法** | 自社リスクモデル + ハザードマップ + 現地調査 |
| **期待する成果** | 衛星データ（降水パターン、地形、地表面温度）による補完的リスク指標 |
| **支払い意思（推定）** | 年間500〜2,000万円（API利用） |
| **重視する指標** | 液状化リスク、土砂災害指定、標高、降水統計 |

#### ペルソナ4: 建設コンサルタント

| 項目 | 内容 |
|------|------|
| **課題** | クライアントへの初回提案で根拠付きの一次調査メモを素早く作りたい |
| **現在の対処法** | GIS + 各種ポータル + 手動レポート作成 |
| **期待する成果** | 提案書の付属資料を半自動生成。出典明記で信頼性を担保 |
| **支払い意思（推定）** | 月額3〜15万円（年間36〜180万円） |
| **重視する指標** | 網羅性、出典の明確さ、レポート出力 |

### ニーズの共通パターン

1. **スピード**: 1件あたりの調査時間を「半日→5分」に短縮したい
2. **統合性**: 複数サイト巡回をワンストップにしたい
3. **根拠性**: 「何を見てそう言ったのか」を明示したい
4. **専門知識不要**: 衛星データやGISの専門知識なしで使いたい
5. **部分的でもOK**: 完璧な評価でなく、初期の足切り判断に使える情報で十分

**最も危険な3仮説**:
1. **A2**: 衛星データが不動産判断材料として認知されるか（価値仮説）
2. **A4**: 衛星データの解像度が地点分析に実用的か（ユーザビリティ仮説）
3. **A1**: 初期スクリーニング短縮への支払い意思があるか（事業性仮説）

## 6. 検証実験の設計

### 実験1: 衛星データの不動産判断価値検証（A1 + A2）

| 項目 | 内容 |
|------|------|
| **仮説** | 不動産デベロッパーの土地仕入れ担当者は、衛星環境データ（標高・NDVI・LST）を初期判断に「有用」と評価する |
| **方法** | コンシェルジュMVP。5〜10名の不動産担当者に、実際の候補地について手動で統合レポートを作成・提示し、フィードバックを得る |
| **成功基準** | 8/10名以上が「初期スクリーニングに使いたい」と回答 |
| **工数** | 2週間（レポート手動作成 + インタビュー） |
| **必要なもの** | 不動産デベロッパーへのアクセス（5〜10名） |

### 実験2: 解像度の実用性検証（A4）

| 項目 | 内容 |
|------|------|
| **仮説** | LST ~5km / NDVI ~5kmの解像度でも「広域コンテキスト」として有用と認識される |
| **方法** | 既知の候補地3箇所について、異なる解像度のデータを見せ、「広域コンテキストとして参考になるか」を評価してもらう |
| **成功基準** | 7/10名以上が「参考になる」と回答。ただし「この解像度では使えない」が3名以上なら要ピボット |
| **工数** | 1週間（データ準備 + インタビュー） |
| **ピボット案** | AW3D標高（~30m）と行政データのみに絞り、LST/NDVIは「参考情報」として扱う |

### 実験3: 支払い意思検証（A7）

| 項目 | 内容 |
|------|------|
| **仮説** | 月額5万円でこのレポートを契約する意思がある |
| **方法** | 実験1のフィードバック後に「月額5万円で無制限利用できるとしたら契約するか」を質問（XYZ仮説: 「不動産デベロッパーの土地仕入れ担当者のうち30%以上が、月額5万円で候補地環境レポートを契約する」） |
| **成功基準** | 30%以上が「契約する」または「上長に相談する」と回答 |
| **工数** | 実験1と同時実施 |

### 実験4: 技術実現性検証（A3 + A9）

| 項目 | 内容 |
|------|------|
| **仮説** | JAXA MCP + Geospatial MCP + DPF MCPの3サーバー同時接続で10秒以内にレスポンスが返る |
| **方法** | 技術プロトタイプ。Node.jsで3 MCP stdio接続を並列実行し、10地点で応答時間を計測 |
| **成功基準** | 80%以上のリクエストが15秒以内に完了 |
| **工数** | 1〜2週間（APIキー取得含む） |

### 実験5: 利用規約検証（A8）

| 項目 | 内容 |
|------|------|
| **仮説** | 国交省API（不動産情報ライブラリ、DPF）の商用利用が可能 |
| **方法** | 利用規約の精読 + 必要に応じて国交省への問い合わせ |
| **成功基準** | 商用利用が明示的に許可、または禁止されていない |
| **工数** | 1週間 |

## 7. バリュープロポジション

### Value Proposition Canvas

#### 顧客の仕事（Jobs to be Done）

| 機能的ジョブ | 感情的ジョブ | 社会的ジョブ |
|-------------|-------------|-------------|
| 候補地の環境・規制・リスクを調査する | 見落としのない判断をしたい | 根拠のある提案で信頼を得たい |
| 複数候補地を比較して絞り込む | 調査の手間を減らしたい | チーム内での意思決定を効率化したい |
| 調査結果をレポートにまとめる | 新しいデータで差別化したい | 先進的な手法を使っていると見せたい |

#### 顧客の痛み（Pains）

1. 情報が5つ以上のサイトに分散（ハザードマップ、不動産情報ライブラリ、Google Maps、JAXA...）
2. 衛星データは専門知識がないと扱えない
3. 1件の調査に半日〜1日かかる
4. 調査の網羅性に自信が持てない（見落としリスク）
5. レポート作成が手作業

#### 顧客の利得（Gains）

1. 1件5分で主要指標を網羅
2. 衛星データという新しい客観的証拠
3. 出典付きで説明責任を果たせる
4. 候補地の足切りが高速化

#### プロダクトの痛み止め（Pain Relievers）

- **ワンストップ統合**: 3つのMCPを1クエリに集約
- **専門知識不要**: AIが衛星データを自然言語で説明
- **自動レポート**: 構造化された出典付きレポート
- **部分結果対応**: 一部データ取得失敗でも他データで回答

#### プロダクトの利得生成（Gain Creators）

- **新しい証拠レイヤー**: 衛星環境データという従来にない客観指標
- **根拠の透明性**: 全データに出典・取得日・検索条件を明示
- **比較機能**: 複数地点を同一基準で並列比較

### 一言バリュープロポジション

> **「衛星データと行政データを1クエリに統合し、候補地の一次調査を半日から5分に短縮する」**

## 8. PLATEAU / GovTechエコシステムとの位置づけ

### PLATEAUとの関係

Project PLATEAUは2025年度に約300都市に拡大（[国交省](https://www.mlit.go.jp/report/press/toshi03_hh_000167.html)）。本プロダクトはPLATEAUの直接的な競合ではなく、**補完的なデータレイヤー**として位置づけ可能。

- PLATEAU: 3D都市モデル（建物形状・高さ・用途）
- TerraScore: 衛星環境データ + 行政データの統合分析

**将来的な統合**: PLATEAUの3Dビューに衛星データをオーバーレイする拡張が考えられる。

### 国交省GovTechエコシステム

```
国交省データ基盤
├── データプラットフォーム (DPF) ─── mlit-dpf-mcp
├── 不動産情報ライブラリ ──────── mlit-geospatial-mcp
├── PLATEAU (3D都市モデル) ──── オープンデータ
├── ハザードマップポータル ──── 国土地理院
└── G空間情報センター ────── データ流通基盤

→ TerraScore は上位2つのMCPを活用し、JAXAレイヤーを追加統合
```

## 12. ハッカソン向けピッチ案

### 30秒ピッチ

> 「候補地の初期調査に半日かけていませんか？TerraScoreは、JAXAの衛星データで地球を見て、国交省のデータで街を知る。座標を入れるだけで、環境・規制・災害リスクを5分で診断します。」

### エレベーターピッチ（1分）

> 「不動産デベロッパーが候補地を評価するとき、ハザードマップ、不動産情報ライブラリ、Google Mapsと最低3つのサイトを巡回し、半日かけて調査しています。
>
> TerraScoreは、JAXA Earth APIの衛星データと国交省MCPの行政データを、MCP（Model Context Protocol）で統合します。座標と半径を入れるだけで、標高・植生・地表面温度の環境コンテキストと、用途地域・地価・災害リスク・周辺施設の行政データが、根拠付きの一次調査レポートとして返ってきます。
>
> 衛星データで「物理的な証拠」を、行政データで「制度的な証拠」を。この二層構造が、既存のどのツールにもない付加価値です。」
