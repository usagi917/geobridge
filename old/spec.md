# TerraScore MVP Spec

更新日: 2026-03-12
元資料: `research.md`
ステータス: MVP仕様ドラフト

## 1. Summary

TerraScore は、JAXA の衛星データと国交省の行政データをまとめて、候補地の一次調査レポートを返す B2B 向けアプリである。

最初に作るべきものは「候補地クイック診断」。不動産デベロッパーの土地仕入れ担当が、候補地の初期スクリーニングを半日から 5 分に短縮できることを狙う。

## 2. Background

候補地の一次調査では、地価、用途地域、都市計画、災害リスク、周辺施設、地形、暑熱や植生などの情報が複数のサービスに分散している。

現状の課題は次の通り。

- 1 件の調査に半日から 1 日かかる
- 複数サイトを手作業で巡回している
- 衛星データは専門知識がないと扱いづらい
- 調査メモや提案資料の作成が手作業
- 見落としへの不安が大きい

今このアプリを作る理由は次の通り。

- JAXA Earth API で衛星データを扱える
- 国交省データを MCP 経由で扱える
- PLATEAU や GovTech 系のオープンデータ活用が拡大している
- 災害リスクや ESG の観点から、立地判断の説明責任が強まっている

## 3. Objective

### 目的

候補地の一次調査を、専門知識なしで短時間に行えるようにする。

### 事業上の狙い

- 不動産デベロッパー向けの明確な業務課題を解く
- 衛星データと行政データの統合を差別化要素にする
- 後続の比較機能や業界特化版につながる土台を作る

### 成功指標

- 1 地点の一次調査レポートを 5 分以内で確認できる
- インタビュー対象の 8/10 が「初期スクリーニングに使いたい」と回答する
- インタビュー対象の 30% 以上が「月額 5 万円なら契約検討可能」と回答する
- 主要 API 呼び出しの 80% 以上が 15 秒以内に完了する

## 4. Market Segment

### 第一ターゲット

不動産デベロッパーの土地仕入れ担当

### このターゲットを優先する理由

- 課題が具体的で頻度が高い
- 時間短縮の価値が分かりやすい
- 初期スクリーニング用途なら MVP の粗さを許容しやすい
- 想定単価が成立しやすい

### 後続ターゲット

- 建設コンサルタント
- 自治体の都市計画、公有地活用担当
- 損害保険会社のリスク評価担当

## 5. Value Proposition

### 一言でいうと

衛星データと行政データを 1 クエリに統合し、候補地の一次調査を半日から 5 分に短縮する。

### 提供価値

- ワンストップで主要な公開情報を確認できる
- 衛星データを自然言語で理解できる
- 出典付きのレポートで説明責任を果たせる
- 完璧な判定ではなく、足切り判断を高速化できる

### 差別化ポイント

- JAXA の衛星データと国交省データを同じ流れで取得する
- 「事実 / 不足 / リスク」を分けた説明型レポートを返す
- MCP ベースのため拡張しやすい
- 高機能 GIS より軽く、既存の業務 SaaS より根拠の幅が広い

## 6. Solution

### 6.1 Product Overview

ユーザーが緯度経度と半径を入力すると、候補地の一次調査レポートを返す。

レポートは次の 2 層で構成する。

- 物理的証拠: JAXA の衛星データ
- 制度的証拠: 国交省の行政データ

### 6.2 Main User Flow

1. ユーザーが緯度、経度、半径を入力する
2. システムが JAXA と国交省のデータを並列取得する
3. 取得したデータを正規化し、地点ごとの結果にまとめる
4. LLM が「事実 / 不足 / リスク」に整理して文章化する
5. UI に数値、要約、出典、取得日時、失敗した項目を表示する

### 6.3 MVP Scope

MVP で必須とする機能は次の通り。

- 緯度経度入力
- 半径入力。ただし上限は 400m
- 候補地 1 点に対する統合レポート表示
- JAXA データ取得
- 国交省 Geospatial MCP データ取得
- 必要に応じた DPF による関連データ導線表示
- 出典、取得日時、検索条件の明示
- 一部取得失敗時の継続表示

### 6.4 Report Sections

レポートは次の順で表示する。

1. サマリー
2. 基本条件
3. 規制と都市計画
4. 災害と安全性
5. 周辺施設と生活利便
6. 衛星環境コンテキスト
7. 不足データと注意点
8. 出典一覧

### 6.5 Data Inputs

#### 入力

- `latitude`
- `longitude`
- `radius_m`

#### 入力制約

- `radius_m` は 1 から 400 の整数
- MVP では住所入力を受けない
- MVP では複数地点の同時比較を受けない

### 6.6 Data Sources

#### JAXA 側

- AW3D: 標高、地形コンテキスト
- LST: 暑熱コンテキスト
- NDVI: 植生コンテキスト
- GSMaP: 降水コンテキスト

MVP では AW3D を主力データとする。LST と NDVI は解像度が粗いため、区画判定ではなく広域参考情報として扱う。

#### 国交省側

- 地価関連
- 都市計画区域
- 用途地域
- 災害危険区域
- 液状化や盛土などの関連情報
- 周辺の公共施設、生活施設
- 将来人口などの補助指標

### 6.7 Output Requirements

各レポートは次を満たすこと。

- 数値と文章の両方を返す
- 断定や推奨ではなく観測事実と注意点を返す
- 出典と取得日が追える
- 欠損データを欠損として明示する
- 「最終判断には追加確認が必要」と分かる表現にする

### 6.8 UX Requirements

MVP の画面は最小でよい。

- 画面 1: 入力フォーム
- 画面 2: レポート画面

レポート画面で必要な要素は次の通り。

- 入力条件の再表示
- セクションごとの見出し
- 重要な数値カード
- 文章サマリー
- 取得失敗項目の表示
- 出典リンクまたは出典名

地図 UI は MVP に含めない。

### 6.9 Functional Requirements

#### FR-1 入力検証

- 緯度経度の形式を検証する
- 半径 400m 超の入力を拒否する

#### FR-2 データ取得

- JAXA、Geospatial MCP、DPF を必要に応じて呼び出す
- 呼び出しは可能な限り並列化する
- タイムアウト時は部分結果で継続する

#### FR-3 データ整形

- API ごとの返却形式を統一する
- 地点単位のレポート構造にマッピングする

#### FR-4 レポート生成

- LLM に渡す前に根拠データを整理する
- 出力は「事実 / 不足 / リスク」の構造を守る
- スコアやランクは付けない

#### FR-5 エラーハンドリング

- データ欠損や API 失敗を利用者に明示する
- 一部失敗でも他のセクションは表示する

#### FR-6 監査可能性

- どのデータを使ったかを追える
- 入力値と取得時刻を表示できる

### 6.10 Non-Functional Requirements

- 初回表示目標は 15 秒以内
- 専門知識がなくても読める文章にする
- スマホ最適化は必須ではないが、最低限読めること
- 失敗時にも空画面にしない
- API キーはサーバー側で管理する

### 6.11 Assumptions

現時点で未検証の前提は次の通り。

- 不動産デベロッパーは衛星データを初期判断に有用と感じる
- LST と NDVI の粗い解像度でも広域コンテキストとして価値がある
- 月額 5 万円以上の課金が成立する
- 国交省 API の商用利用に問題がない
- 3 系統のデータ取得が実用速度で動く

## 7. What We Will Not Build in MVP

MVP では次をやらない。

- 住所ジオコーディング
- 地図 UI
- PDF 出力
- 複数地点比較
- 自動スコアリング
- 用地の推奨判定
- 売上予測、価格予測
- 法的適合判断
- ユーザー認証
- 決済
- 自動監視や通知

## 8. Risks and Mitigations

### リスク 1: 衛星データの解像度不足

LST と NDVI は地点レベルの判断に粗い可能性がある。

対応:

- AW3D を主要な衛星指標にする
- LST と NDVI は参考情報と明記する
- 検証で不評なら MVP から外せる設計にする

### リスク 2: 国交省 MCP の不安定さ

α版 API への依存がある。

対応:

- 取得失敗に強い設計にする
- REST API へのフォールバック余地を残す

### リスク 3: LLM レポートへの不信感

説明文が過剰に断定すると信頼を損なう。

対応:

- 事実と推測を分ける
- 出典と欠損を明示する
- 推奨や断定を避ける

### リスク 4: 商用利用条件

API やデータの利用条件が制約になる可能性がある。

対応:

- 実装前に利用規約を再確認する
- 必要なら非商用利用や PoC に範囲を絞る

## 9. Release Plan

### Phase 0: Discovery

- 不動産デベロッパー 5 から 10 名にコンシェルジュ MVP を実施
- 支払い意思を確認する
- 解像度の実用性を確認する
- API 利用規約を確認する

### Phase 1: MVP

- 1 地点入力
- テキスト中心の統合レポート
- 出典表示
- 部分失敗対応

### Phase 2: Expansion

- 複数地点比較
- 災害リスク特化モード
- 物流、DC 向けの業界特化版
- PLATEAU 連携

## 10. Implementation Notes

### 推奨構成

- フロントエンド: 単一入力画面 + レポート表示画面
- バックエンド: MCP オーケストレーション層
- データ処理: API 応答の正規化層
- レポート生成: LLM による説明生成層

### 実装で先に着手すること

1. 国交省 API キーの取得
2. JAXA Earth API の最小取得確認
3. 3 系統データ取得の並列実行プロトタイプ
4. レポート JSON スキーマの定義
5. 最小 UI の作成

### レポート JSON の最小構造

```json
{
  "input": {
    "latitude": 35.0,
    "longitude": 139.0,
    "radius_m": 400
  },
  "summary": {
    "facts": [],
    "gaps": [],
    "risks": []
  },
  "land": {},
  "planning": {},
  "hazard": {},
  "facilities": {},
  "satellite": {},
  "sources": [],
  "errors": []
}
```
