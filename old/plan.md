# TerraScore MVP Plan

更新日: 2026-03-12

## 1. インタビュー要約

- 想定ユーザーは `個人の引越し検討者` と `自治体`
- `Qwen` には `MCP呼び出しの補助`、`レポート生成`、`チャットUI/UXの中核` を期待する
- まずは `MVP` を作る
- 出力したい価値は `土地の価値 / 災害 / 住みやすさ` を網羅したレポート

## 2. 壁打ちの結論

このまま `個人向けアプリ` と `自治体向け業務ツール` を同時に作ると、MVPが広がりすぎる。

そのため、最初の勝ち筋は次の形に絞る。

- `使う人`: 引越しや移住を検討している個人
- `MVPで向き合う顧客`: 個人
- `Phase 2 の展開先`: 自治体の移住促進、定住支援、地域広報部門
- `提供価値`: 候補エリアについて、災害リスク、生活利便、土地・地域コンテキストを根拠付きで短時間に理解できる

つまり、MVPは `個人向けの居住地デューデリジェンスAI` として設計し、自治体展開は後段に回す。

この形なら、あなたがやりたい `個人向けの体験` を先に成立させつつ、あとから `自治体に役立つ価値` に拡張しやすい。

## 3. プロダクト定義

### 一言でいうと

`Qwen` と `JAXA / 国交省MCP` を組み合わせ、候補地の「住む価値」を対話的に説明するレポートAI。

### 解く課題

- 個人は、引越し候補地の災害、安全、生活利便、地域性を横断比較しづらい
- 自治体は、移住検討者に対して地域の良さと注意点を根拠付きで説明しづらい
- 公開データは存在するが、分散しており、読むだけで負荷が高い

### MVPで約束する価値

- 1つの候補地について、`災害`, `暮らしやすさ`, `土地・地域の文脈` を1回で確認できる
- すべての主張に `出典` と `取得日時` を付ける
- `断定` ではなく `判断支援` に徹する
- `足りないデータ` も明示する

## 4. MVPの対象ユーザー

### Primary User

- 子育て世帯を含む引越し・移住検討者

### Phase 2 Buyer / Partner

- 移住促進を行う自治体

### この順番にする理由

- チャットUIとレポートUXは個人向けの方が価値が伝わりやすい
- 自治体は「住民向けの説明ツール」として後から再利用しやすい
- 不動産評価や投資判断よりも、公共性と納得感の文脈で出しやすい
- B2C と B2G を同時に検証すると、成功指標と営業動線がぶれる

## 5. MVPで扱う価値の定義

`土地の価値` は、価格予測や鑑定ではなく、次の4軸で扱う。

1. `災害・安全性`
2. `日常生活の利便性`
3. `自然環境・暑さ・緑地などの居住環境`
4. `公的データから読める地域コンテキスト`

MVPでは `資産価値の予測` はやらない。
代わりに、`地価`, `用途地域`, `将来人口`, `周辺公共施設` のような proxy を示す。

## 6. MVPで作る体験

### 入力

- ユーザーが `住所・地点・エリア名` のいずれかを入力する
- あわせて、`子育て重視`, `災害重視`, `生活利便重視` のような閲覧観点を選べる

### 出力

1. 総合サマリー
2. 災害・安全性
3. 暮らしやすさ
4. 自然環境
5. 地域コンテキスト
6. 不足データと注意点
7. 出典一覧

### MVPで必須

- チャット入力
- 1地点レポート生成
- JAXA と国交省MCPの呼び出し
- 根拠付き要約
- 取得失敗時の部分表示
- 出典と取得日時の表示

### MVPでやらない

- 物件価格の予測
- 売買の推奨
- フルGIS編集機能
- 3地点以上の比較
- PDF帳票の作り込み
- リアルタイム災害通知

## 7. Qwenの役割

`Qwen` は何でも直接決める AI ではなく、次の3役に限定する。

### 1. Query Interpreter

- ユーザーの質問から、必要な調査観点を抽出する

### 2. Report Writer

- 正規化済みデータを、読みやすい自然言語レポートに変換する

### 3. Chat Guide

- 「次に何を見るべきか」「どこが不確実か」を会話で補助する

### 役割を持たせないもの

- 生データを根拠なく解釈して断定すること
- API失敗時に事実を補完して埋めること
- 不動産価格や災害確率を勝手に予測すること

## 8. システム構成方針

### 基本構成

1. `Chat UI`
2. `App Server`
3. `MCP Orchestrator`
4. `JAXA Earth API / MLIT Geospatial MCP / MLIT DPF MCP`
5. `Normalization Layer`
6. `Qwen Report Layer`
7. `Citation / Cache Layer`

### 設計上の重要方針

- MCP呼び出しは `Qwenに丸投げしない`
- ツール実行はサーバー側で許可制にする
- 返却値は必ず構造化してから Qwen に渡す
- Qwen には `事実`, `不足`, `注意点` だけを書かせる
- 出典URL、取得時刻、失敗項目は LLM の外で保持する

## 9. データソースの役割分担

### JAXA

- 標高
- 地表面温度
- 植生
- 降水

役割:
行政データでは見えにくい `物理環境` と `広域文脈` を補う。

### 国交省 Geospatial MCP

- 地価
- 用途地域
- 災害関連情報
- 生活関連施設
- 将来人口などの地域情報

役割:
`制度`, `インフラ`, `生活`, `リスク` の説明根拠を担う。

### 国交省 DPF MCP

- 補助データ探索
- PLATEAUや関連データへの導線

役割:
MVPの主役ではなく、`補足調査` のための拡張レイヤーとして使う。

## 10. 開発フェーズ

### Phase 0: 問いの固定

- 1つのユースケースに絞る
- レポートの見出しを固定する
- データ項目ごとの「言ってよいこと / 言ってはいけないこと」を決める

### Phase 1: 技術スパイク

- JAXA の取得項目を 2-3 個に絞って疎通確認
- MLIT Geospatial MCP の主要APIを 3-5 本に絞る
- 単一地点の JSON を正規化できる形にする
- Qwen で根拠付きレポートが安定生成できるか確認する

### Phase 2: MVP実装

- チャット入力画面を作る
- 1地点レポート画面を作る
- 出典表示と失敗表示を実装する
- 重視観点に応じて要約トーンを変えられるようにする

### Phase 3: パイロット

- 実際の引越し検討者に触ってもらう
- どのセクションが意思決定に効いたかを測る

### Phase 4: 自治体展開の検証

- 個人向けで有効だったレポートを自治体の移住促進文脈に再設計する
- 自治体向けの説明テンプレートや地域紹介テンプレートを作る
- 実証先を1自治体に絞る

## 11. 成功指標

### プロダクト指標

- 1地点レポートが `20秒以内` に返る
- 主要セクションの `出典表示率 100%`
- API失敗時も `空画面にしない`

### ユーザー価値指標

- テストユーザーの過半数が「候補地理解が早くなった」と回答する
- レポート後に追加質問が自然に発生する

### Phase 2 指標

- 自治体担当者が「住民説明や移住相談に使える」と回答する

## 12. 最大リスクと対策

### リスク1: スコープが広すぎる

対策:
`1地点`, `1レポート`, `4-5セクション` に固定する。

### リスク2: 個人向けの入力体験が重い

対策:
最初は `住所入力` にこだわりすぎず、`自治体名 + エリア名` や `地図ピン` でも成立するUIにする。

### リスク3: LLMがもっともらしく言い過ぎる

対策:
テンプレートを `事実 / 不足 / 注意点` に固定し、推奨表現を禁止する。

### リスク4: 自治体向けに売るには時間がかかる

対策:
MVPでは自治体販売を狙わず、まずは個人向けの価値を検証する。

### リスク5: 価値の定義が曖昧

対策:
`土地の価値` ではなく、`住む判断に必要な情報の質` を磨く。

## 13. 直近の実行順

1. MVPのユースケースを `移住・引越し候補地の1地点診断` に固定する
2. 入力形式を `住所`, `エリア名`, `地図ピン` のどれで始めるか決める
3. JAXA と MLIT から使う最小データ項目を決める
4. 正規化JSONのスキーマを作る
5. Qwen に渡すレポートテンプレートを作る
6. チャット画面とレポート画面の2画面だけ作る
7. 5-10件の代表プロンプトで品質検証する
8. 引越し検討者に見せてフィードバックを取る
9. 価値が出たら自治体向け再パッケージを検討する

## 14. 最終判断

このアイデアは成立するが、`土地の価値を全部見るAI` として出すと広すぎる。

MVPでは、`引越し検討者向けの居住地レポートAI` に絞るべきである。

自治体は有望な展開先だが、同時に追うのではなく `Phase 2` に回す。

この切り方なら、`Qwen × MCP` の強みが分かりやすく、JAXA と国交省データの組み合わせにも必然性が出る。

## 15. 事業化の補足

もし最優先が `売上化` であれば、個人向けよりも `不動産・金融・調査業務のスクリーニング用途` の方が初期顧客としては強い可能性が高い。

ただし今回は、あなたの回答どおり `MVPをまず作る` ことを優先し、かつ `チャットUI/UX` を強みにしたい前提なので、まずは `個人向けの体験価値` を検証する計画にしている。
